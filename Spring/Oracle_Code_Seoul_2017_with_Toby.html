@Controller
public class HelloWorldController{
    @RequestMapping("/helloWorld")
    public String helloWorld(Model model){
        model.addAttribute("message", "Hello World!");
        return "helloWorld";
    }
}

# @Controller

  • 스프링 빈 스캐너에 의해서 싱클톤 빈으로 등록
  • 빈 이름을 지정하지 않아도 클래스명의 제일 앞글자만 소문자로 바꿔서 helloWorldController으로 빈이름 지정
  • MVC 컨트롤러 기능 담당
  • @Controller가 붙은 클래스에 @RequestMapping가 있으면, 이것을 맵핑정보로 활용해서 웹컨트롤러 메소드로 사용하게 해준다.

# @RequestMapping("/helloWorld)

  • 웹 요청 /helloWorld의 처리를 담당하는 메소드
  • 모든 HTTP Method , 파라미터, 헤더 허용(GET, POST, PUT이든 다 허용)
  • 타입 레벨 @RequestMapping이 있으면 4가지 조건으로 결합

# model.addAttribute("message", "Hello World!");

  • 파라미터에 Model 타입이 있으면 뷰렝 전달하는 모델 정보를 담는데 이용 가능

# return "hello World";

  • 리턴 타입이 String이면 뷰 리졸버에서 사용할 뷰 네임으로 사용
  • 단, 메소드에 @ResponseBody가 있으면 컨테이너에 등록된 MessageConverter를 이용해 응답 바디로 변환

# 애노테이션과 메타프로그래밍, 관례의 범람

애노테이션의 한계와 부작용

  • 컴파일러에 의해 검증 가능한 타입이 아님
  • 코드의 행위를 규정하지 않음
  • 상속, 확장 규칙의 표준이 없음
  • 이해하기 어려운 코드 , 오해하기 쉬운 코드 가능성
  • 테스트가 어렵거나 불가능함 (애노테이션은 테스트가 안된다, 프레임워크에 올려야만 기능이 동작하게 되어있어서 손쉽게 확장도 힘들고 추상화된 방식으로 매핑을 할 수 없음.. 만들어 준걸 사용하는 수준에서 머문다)
  • 커스터마이징하기 어려움

# 이해안되던 코드

@Configuration
public class MyController{
    @Bean MyService myService(){
        return new MyService();
    }
    
    public void run(){
        System.out.println(myService() == myService());
    }
    
    public static void main(String[] args){
        new MyController().run();
        
        ApplicationContext ac = new AnnotationConfigApplicationContext(MyController.class);
        MyController mc = ac.getBean(MyController.class);
        mc.run();
    }
}
